# product-lifecycle-templates: Repository Overview

> As of 2026-07-10. This document is a single-pass synthesis for anyone who needs a complete picture of what this repo is, where it stands, and what comes next.

---

## 1. Vision

The north-star is simple and demanding: **be the indisputable best-in-class reference implementation of a template library**, not a folder of templates.

The specific positioning: *the template layer that humans and AI agents both reach for first - governed, self-describing, sized to context, and traceable. Coverage is not the product; correct-by-construction reuse is.*

The vision rests on three simultaneous claims:

1. **Quality over coverage.** Any PM can publish a folder of `.md` files. The differentiated bet is that every artifact in this library is researched, dual-readable, correctly sized, and proven to work - backed by a CI-enforced Definition of Done rather than aspiration.

2. **Agent-native from the start.** Templates ship with machine-readable metadata (`pairs_with` skill IDs, `sizes_available`, `doc_type`, lifecycle `phase`) so an AI agent can select, fetch, fill, and validate deterministically. The library is designed for LLM consumers, not just human ones.

3. **Ecosystem coherence with pm-skills.** This repo is the deliberate sibling of `pm-skills` (68 skills, mature CI, plugin/marketplace distribution). The division is clean: a skill teaches an agent *how to produce* an artifact; a template is *the artifact itself*. The `pairs_with` field in each bundle's metadata is the seam. Two repos form a loop a single repo cannot.

The full-length aspiration is documented in [`template-library-design-spec.md`](_local/initial-discovery/docs/template-library-design-spec.md) (§1-§3, §16 roadmap) and articulated in the README's one-line moat statement.

## 2. What a "Bundle" Is

The primary unit of the library is a **bundle** - a folder containing eight files for one document type. The bundle is the thing that earns trust; the blank template is only one file in it.

| File | Role |
|---|---|
| `<type>_template-lean.md` | Blank shape, minimum useful; carries enriched HTML comment guidance per section |
| `<type>_template-full.md` | Blank shape, comprehensive; strict superset of lean (nesting rule) |
| `<type>_companion.md` | Deep explainer: what it is, why, debates, 11-section skeleton, tiered citations |
| `<type>_guide.md` | Operator card: when-to-use, quality rubric, named anti-patterns |
| `<type>_example.md` | Fully worked instance, no lorem, no fabricated metrics |
| `<type>_meta.yaml` | Machine manifest: catalog metadata, `pairs_with`, `sizes_available` |
| `<type>_history.md` | Per-bundle changelog by template version |
| `<type>_research-log.md` | Evidence trail: every source, its tier, what it supports |

The **nesting rule** is the structural heart of the sizing model: lean section IDs are a strict subset of full section IDs, in the same order. A document can grow from lean to full without a re-author.

**Section guidance standard** (the enriched comment format, decided 2026-06-30): each section in a blank template carries an HTML comment with WHAT / WHY (deep-linked to the companion) / ASK / GOOD / WEAK / TRAP. A "How to fill this in" preamble opens each template. Guidance vanishes on render; the blank shape stays clean.

**Citation standard** (methodology v0.2.0): every non-obvious companion claim carries a numbered, reliability-tagged citation (`[primary]` / `[practitioner]` / `[vendor]`). Citations are hyperlinked (`[[n]](#ref-n)` anchors), not bare text.

## 4. Strengths

**1. Genuine research backing.** Every companion document was produced via the methodology's research protocol: live web searches, tiered sources (`[primary]`/`[practitioner]`/`[vendor]`), claim-by-claim verification, contested-point acknowledgment. The PRD companion alone cites 13 sources with hyperlinked anchors. Most template collections cite nothing.

**2. The dual-reader split.** The companion / guide separation (modeled on Diataxis explanation/how-to) is a real differentiator. A beginner gets a 2,000-5,000-word deep explainer with lineage, debates, and anti-patterns. A practitioner in a hurry gets a one-page operator card with a quality rubric. Neither reader has to wade through material for the other.

**3. Enriched comment guidance.** The HTML comment standard (WHAT / WHY / ASK / GOOD / WEAK / TRAP) makes the blank template a self-contained coaching tool. The guidance vanishes on render - no reader sees scaffolding in the finished doc.

**4. The nesting rule enforced.** CI confirms that lean sections are a strict ordered subset of full sections. Upgrade-in-place is guaranteed structurally, not by convention.

**5. The coherent example family.** All four examples chain on one "Saved Views" scenario. This is unusually disciplined; most template repos have disconnected examples that authors can't cross-reference.

**6. Machine-readable metadata.** `_meta.yaml` per bundle carries `pairs_with`, `sizes_available`, `doc_type`, `phase`, `family`, `status`, `template_version` - everything an agent needs to select, validate, and fill a template deterministically.

**7. The Product Artifact Atlas.** A self-contained interactive HTML file mapping all 205 catalog types, filterable by tier, phase, and category, with relationship data. No comparable navigable map of the PM/SDLC artifact space exists publicly.

**8. The governance gate is automated.** `check-bundles.py` runs 6 structural checks in one command. All 4 bundles pass. This is a local prototype, but the principle is enforced - DoD is not aspirational.

**9. Honest documentation of decisions.** The strategy briefs and implementation plan record not just what was decided but why, what alternatives were considered, and what remains open. The repo is its own design-history file.

## 5. Weaknesses

**1. No distribution or installation story.** There is no `.claude-plugin/plugin.json`, no `marketplace.json`, no `catalog.md`, no `manifest.json` at the repo root. The library cannot be installed by any consumer - human or agent - except by cloning the repo. The `pairs_with` seam to pm-skills is metadata-level only; nothing in the system actually uses it yet.

**2. Work lives in `_local/` - the provisional home was never graduated.** The `_local/` path was explicitly declared temporary (scaffold deferred, phase in metadata). The proper top-level `templates/` directory from the design spec does not exist. No production consumer would know to look in `_local/`.

**3. ADR files were never written.** The four foundational decisions (name, variant model, phase vocabulary, first family) are documented in the strategy brief and implementation plan, but the dedicated `docs/decisions/` ADR files the plan required do not exist. Future contributors have to mine prose to find these decisions.

**4. CI is local Python, not GitHub Actions Node.js.** The plan called for 7 Node.js scripts in `scripts/` wired to `.github/workflows/ci.yml` (the pm-skills pattern). What exists is a single Python script in `_local/tools/`. It works locally but is not enforced on push, is not in the language the plan committed to, and is not at the path the plan specifies.

**5. No family contract file.** `_families/delivery-docs.contract.md` was listed as a P6 output and not written. The shared section vocabulary, required fields, and nesting expectation for the delivery-docs family are not formally documented; they exist tacitly in the methodology and individual bundles.

**6. "Best-in-class" is asserted, not measured.** Every quality gate checks structure (files present, nesting holds, citations resolve, no dashes). Not one check asks whether filling a template actually produces a better document. The sibling pm-skills repo has an `evals/` directory with behavioral tests; this repo has none. The entire value proposition rests on a claim that can't currently be substantiated.

**7. No template has ever been filled.** Zero real users. The methodology says a bundle "survives one usage cycle" is the adoption threshold; no bundle has cleared it. There is no feedback loop from actual use back to improvement.

**8. Single-domain examples only.** All four examples chain on one B2B-SaaS feature ("Saved Views"). The raising-the-ceiling brief (CT-1) flags this: a best-in-class library should demonstrate its templates travel across domains (regulated/health, consumer-mobile, platform/API). Coverage of one scenario proves the template works for one scenario.

**9. Open decisions stalling distribution.** D2 (does the skills CLI install a template-only repo with no SKILL.md?) has never been tested. This determines whether the `use-template` companion skill is required for the distribution story. It has been open since 2026-06-29 and costs 30 minutes to answer.

**10. No maintenance cadence.** Citations will rot. IEEE 829 was already superseded by ISO/IEC/IEEE 29119 when the catalog was written. QMSR changes went into effect 2026-02-02. The raising-the-ceiling brief's FR-1 (automated staleness gate) is not built. Who re-checks sources, how often, funded how - these questions are unanswered.

**11. Business model silence.** The planning documents say nothing about monetization or sustainability. The raising-the-ceiling brief (Theme G) flags this as a strategic blind spot: the library is an asset to `product-on-purpose`, but whether it is a credibility/funnel asset, an open-core revenue play (generator + regulated packs), or a pure craft project is undecided. That decision shapes what gets built next (MCP, evals, regulated tier) and at what priority.

**12. Solo-maintainer surface.** 205 catalog types at 8 files each would be 1,640 files. With no team, no automation, and no demand-gated pull queue, scale pressure will arrive before bandwidth does.

## 6. Competitive Landscape

No direct competitor ships all of: research-backed companions, lean/full sizing under a nesting rule, enriched comment guidance, machine-readable agent-selectable metadata, and a CI-enforced Definition of Done. The nearest alternatives are:

### 6.1 Curated lists ("awesome" repos)

`dend/awesome-product-management`, `lorabv/awesome-agile`, `nickmchardy/awesome-product-management` and similar: aggregated links to scattered resources. No quality gate, no worked examples, no sizing, no agent-readable metadata. They point outward; this library is the thing they point to.

### 6.2 Platform template galleries (Notion, Confluence, Coda)

Notion's template gallery and Confluence's Blueprint templates are the closest in spirit. Weaknesses: platform-locked, not version-controlled, no research backing, no sizing variants, no companion explainers, and vendor-motivated (every template sells the platform). Portability is zero; Markdown-first is zero.

### 6.3 Tool-vendor template packs (Atlassian, Aha!, ProductPlan)

Built for and locked to specific tools. Useful for their respective user bases; structurally incompatible with a local-first, file-based, agent-readable approach.

### 6.4 pm-skills own templates (`skills/{skill}/references/TEMPLATE.md`)

The most direct predecessor. Each skill bundles its own `TEMPLATE.md`. Weaknesses the design spec names explicitly: not independently addressable, single-variant, carry no self-description, have no provenance. This library supersedes them as an optional, non-breaking enrichment.

### 6.5 Single-artifact exemplars

- Nygard's ADR format (a single artifact, not a library)
- Basecamp's Shape Up templates (methodology-specific, bundled with the book)
- Marty Cagan / SVPG's product spec guidance (prose, not a fillable template)
- Google SRE books' runbook/postmortem formats (prose exemplars, not structured templates)

All are high-quality at the individual artifact level; none is a governed, multi-type library with sizing and CI.

### 6.6 AI-native template products (emerging)

Claude.ai artifacts, ChatGPT's file-generation, and similar LLM-native surfaces that produce documents on demand. These are the future competition and the reason the agent-native moat (`pairs_with`, machine-readable metadata, MCP server) is strategic, not optional. Currently these systems generate documents from training data with no library-level governance or provenance; this library would be the governed substrate they call.

### 6.7 The real differentiation

This library occupies an empty cell: **local-first, version-controlled, research-backed, agent-selectable, CI-governed, dual-readable templates across the full PM/SDLC arc**. The moat is not any one of those properties - it is all of them together, enforced by a Definition of Done rather than asserted by a README.

### 7.2 Strategic next steps (from raising-the-ceiling brief)

Priority ordering from the brief's prioritization table, filtered to the three highest-leverage items:

**LP-2 (grade-my-existing-doc) - the adoption wedge.** Let a user paste a document they already have and get it scored against the bundle's rubric, with specific gaps flagged. Most teams will not start from a blank template, but they will happily have their existing PRD critiqued. This is the top-of-funnel move. Effort: medium. Strongly recommended as the first feature beyond content.

**EV-1 (efficacy evals).** Build a behavioral harness mirroring pm-skills' `evals/`: give a model the blank template plus a scenario, score the filled result against the rubric, compare against a baseline (no template). This is what converts "best-in-class" from a claim to a measurement. It is cheapest to build with the first bundle (now) and most expensive to retrofit across many bundles later. The one item that should not be deferred.

**AG-2 (templates as MCP server).** An MCP server that lets any agent list templates, select by intent (embeddings over purpose/when-used/aliases), fetch the blank plus companion, and post back a filled draft. This is the difference between "a folder humans clone" and "the template layer every agent calls." Required for the agent-native claim to be real. Effort: high; timing: after P5 distribution.

**LP-1 (use-template flow).** A skill or CLI flow that interviews the author section by section (the enriched comments are already the interview script), emits a filled draft with provenance frontmatter, and strips guidance. Resolves open decision D2. Immediate predecessor to AG-2. Effort: medium-high; timing: right after P5.

## 8. Planned Catalog

### 8.1 What's built

4 bundles (delivery-docs family): `prd`, `user-stories`, `acceptance-criteria`, `release-notes`.

### 8.2 Tier 1: must-have 28 (the actual product)

Per the master catalog's TL;DR, these 28 types cover roughly 80% of organizations. Build order within Tier 1 is demand-gated (pull-queue), not sequential. The 24 not yet built:

**Strategy / Vision:** Product Vision, Product Strategy, OKRs, Product Roadmap, Business Case

**Discovery / Research:** User Research Report, User Persona, Customer Journey Map

**Requirements / Definition:** Epic, Definition of Done, Software Design Document / Technical Design Doc

**Decision Records:** Architecture Decision Record (ADR)

**Planning:** Release Plan, Product Backlog (living, not a static template)

**Testing:** Test Plan, Test Case

**Delivery / Operations:** Bug Report (issue template), Deployment Runbook

**Post-release:** Postmortem / Incident Review, Status Report / Weekly Update

**Retrospectives / Ceremonies:** Retro Notes, Sprint Review Notes

**Analytics / Measurement:** Dashboard / Metrics Report spec, RAID / Risk Register

**Design:** Wireframe (format guide, not a Figma template)

**Positioning:** Positioning Statement

Note: some of these (Product Backlog, Dashboard) are living artifacts rather than one-time documents; their template shape is a schema definition rather than a blank-to-fill form. These require a variant-model decision before authoring.

### 8.3 Tier 2: methodology + discipline packs (pull-gated)

Build only when a team is actively practicing the methodology. Candidate families:

- **scrum-docs:** Sprint Goal, Sprint Review notes, Retrospective, Definition of Ready
- **shape-up-docs:** Pitch, Breadboard, Hill Chart, Betting Table agenda
- **amazon-working-backwards:** PR/FAQ, Six-Pager, WBR narrative
- **ux-discovery:** Interview Guide, Survey Instrument, Concept Test Brief, Opportunity Solution Tree
- **analytics:** Experimentation Design, A/B Test Report, Metric Tree definition
- **release-ops:** Rollout Plan, Feature Flag Spec, Incident Runbook

### 8.4 Tier 3: regulated artifacts (separate module, out of scope for now)

FDA design controls (Design History File, Risk Management File per ISO 14971), SOC 2 (System Description, Control Matrix), GDPR (Records of Processing Activities, DPIA). These require traceability-first authoring, current-regulation verification (QMSR effective 2026-02-02), and separate access controls. Not part of the main library.

### 8.5 The atlas as a scoping tool

[`_local/atlas/atlas.html`](_local/atlas/atlas.html) renders all 205 types with tier, phase, category, and relationship data. It is designed to make saying "we deliberately did not build this" a visible, first-class act - the opposite of treating 205 as a backlog.

## 9. Key Architecture Decisions (locked)

| Decision | Choice | Rationale |
|---|---|---|
| **Name** | `product-lifecycle-templates` everywhere | Broader name ages better if library spans beyond PM; drop `pm-` prefix |
| **Variant model** | `lean` / `full` with descriptive filenames; `s/m/l` only where type earns 3 weights | Type decides its variant count; descriptive names are self-documenting; nesting rule enforced |
| **Phase vocabulary** | Lowercase (`deliver`, `discover`, `define`, etc.); bundle IDs `<phase>-<doctype>` | Exact match to pm-skills frontmatter; `pairs_with` resolves against real skill IDs |
| **First family** | `delivery-docs` (`prd`, `user-stories`, `acceptance-criteria`, `release-notes`) | All four `pairs_with` targets already exist in pm-skills |
| **Directory spine** | Document type only; no lifecycle-phase folders | Phase in metadata; avoids combinatorial explosion (size x phase x audience = 45 dirs per type) |
| **Generator (Layer 1)** | Deferred until usage evidence (D1 open) | Speculative generator before real usage inverts the build order |
| **pm-skills relationship** | Additive, non-breaking; skills keep their own TEMPLATE.md | Decoupling (DRY via URI) was rejected to preserve self-containment |

## 11. What to Do Next (if picking one thing)

If only one thing happens, it should be **LP-2 (grade-my-existing-doc)**: the reverse-template flow that lets a user paste a document they already own and get it scored against the bundle's quality rubric. Reasons:

- It requires zero new bundle authoring - the rubrics already exist in the four guide files.
- It is the strongest adoption wedge: most teams won't start from blank, but everyone wants their existing PRD critiqued.
- It generates the first real feedback about what's actually weak in real documents, which is the signal that should drive Tier 1 sequencing.
- It is demonstrable to a skeptic in a five-minute conversation: paste a PRD, get specific gaps back.

The infra work (P2/P3/P5) is necessary for the long-term distribution story, but LP-2 creates pull for the library before any of that ships.
